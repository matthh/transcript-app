## Critical

### C-1 — Unauthenticated transcript write/delete/rename endpoints

**Files:**
- `src/app/api/transcripts/[episode]/route.ts` (PUT line 43, DELETE line 80)
- `src/app/api/transcript-rename/route.ts` (POST line 12)

**Issue:**  
`PUT /api/transcripts/[episode]` overwrites any transcript in Vercel Blob storage.  
`DELETE /api/transcripts/[episode]` deletes any transcript.  
`POST /api/transcript-rename` moves any transcript.  
None of these routes check any form of authentication. Any client that can reach the deployment can corrupt or destroy all transcript data.

**Fix:**  
Add a Bearer token check (reuse `podreview-auth.ts#checkAuth` or a new `ADMIN_TOKEN` env var) at the top of each handler:

```ts
// At the top of the handler:
import { checkAuth } from '@/lib/podreview-auth';
if (!checkAuth(request)) {
  return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
}
```

Or introduce a dedicated `ADMIN_TOKEN` env var and constant-time compare (see `external-auth.ts` for the pattern).

### C-2 — AssemblyAI webhook has no HMAC signature verification

**File:** `src/app/api/transcribe/webhook/route.ts` (POST handler, line 22)

**Issue:**  
The webhook endpoint accepts any POST request claiming a `transcript_id` is completed. AssemblyAI signs webhook payloads with a shared secret (`webhook_auth_header_name` / `webhook_auth_header_value`). Without verifying the signature, an attacker who discovers the webhook URL can:
- Trigger false "completed" saves with crafted transcript bodies.
- Overwrite legitimate transcripts with garbage data.

**Fix:**  
Add a `ASSEMBLYAI_WEBHOOK_SECRET` env var. In the handler, validate the incoming signature before processing:

```ts
const incomingAuth = request.headers.get('authorization'); // or custom header
if (!incomingAuth || !timingSafeEqual(
  Buffer.from(incomingAuth),
  Buffer.from(process.env.ASSEMBLYAI_WEBHOOK_SECRET ?? '')
)) {
  return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
}
```

Alternatively, configure AssemblyAI to include an `Authorization` header with a shared secret and validate it server-side.

## High

### H-1 — Unauthenticated transcription trigger endpoint

**File:** `src/app/api/transcribe/route.ts` (POST handler, line 19)

**Issue:**  
`POST /api/transcribe` starts an AssemblyAI transcription job for any audio URL supplied in the request body. There is no authentication. An attacker can:
- Submit arbitrary audio URLs (including external URLs, enabling SSRF against internal services if running on a private network).
- Exhaust AssemblyAI credits by triggering many jobs.

**Fix:**  
Add Bearer token auth (reuse `ADMIN_TOKEN` or `PODREVIEW_PASSWORD` pattern).  
Additionally, validate that `audioUrl` is a Vercel Blob URL (matches `https://*.vercel-storage.com/`) before passing it to AssemblyAI to mitigate SSRF:

```ts
const ALLOWED_AUDIO_HOST = /^https:\/\/[^/]+\.vercel-storage\.com\//;
if (!ALLOWED_AUDIO_HOST.test(audioUrl)) {
  return NextResponse.json({ error: 'Invalid audio URL' }, { status: 400 });
}
```

### H-3 — `podreview-auth.ts` uses non-constant-time string comparison

**File:** `src/lib/podreview-auth.ts` (line 6)

```ts
return auth.slice(7) === process.env.PODREVIEW_PASSWORD;
```

**Issue:**  
JavaScript's `===` on strings is not constant-time; it short-circuits on the first differing byte. In theory, a remote timing oracle can enumerate the password character by character. The window is small (network jitter dominates) but the fix is trivial.

**Fix (low-risk, applied in this PR):**

```ts
import { timingSafeEqual } from 'crypto';

export function checkAuth(request: NextRequest): boolean {
  const auth = request.headers.get('authorization');
  if (!auth?.startsWith('Bearer ')) return false;
  const provided = auth.slice(7);
  const expected = process.env.PODREVIEW_PASSWORD ?? '';
  if (provided.length !== expected.length) return false;
  return timingSafeEqual(Buffer.from(provided), Buffer.from(expected));
}
```

### M-4 — In-memory rate limiter resets on cold start / does not span instances

**File:** `src/lib/external-rate-limit.ts` (line 1–8 comment)

**Issue:**  
The file documents this as a known stopgap. On Vercel's serverless edge, each cold start resets all counters. A high-traffic period that triggers many cold starts will effectively disable the rate limiter for bursts. With the current single external consumer (Explore) this is low risk, but adding any consumer increases the exposure.

**Fix:**  
Migrate to Upstash Redis or Vercel KV for accurate cross-instance rate limiting. Until then, document the limitation in README/env-var table.

## Low

### L-1 — `/api/coverage` and `/api/speakers` do unbounded parallel fetches

**Files:**
- `src/app/api/coverage/route.ts` (line 142 — `Promise.all(fetches)`)
- `src/app/api/speakers/route.ts` (inner loop fetching each transcript)

**Issue:**  
Both endpoints issue one `fetch()` call per transcript blob, all in parallel. With 300+ transcripts this creates ~300 concurrent network requests per invocation. This can saturate Vercel's connection pool and cause intermittent fetch failures.

**Fix:**  
Use a concurrency-limited batch (e.g., `p-limit` or a simple manual queue with a window of 20-30) in both endpoints.

### L-2 — Feedback email renders user-supplied content as raw HTML

**File:** `src/app/api/feedback/route.ts` (line 136–181)

**Issue:**  
`entry.name`, `entry.query`, `entry.answer`, and `entry.comment` are interpolated directly into an HTML email body with no escaping. A malicious user submitting `<script>` tags or HTML injection in a query or comment could craft a feedback email that exploits the email client of the recipient. The risk is low (email clients generally sandbox scripts), but it violates standard hygiene.

**Fix:**  
HTML-escape all user-supplied fields before interpolating into the email template. A simple `escapeHtml()` helper covers the cases:

```ts
function escapeHtml(s: string): string {
  return s.replace(/&/g, '&amp;').replace(/</g, '&lt;').replace(/>/g, '&gt;').replace(/"/g, '&quot;');
}
```

Apply to `entry.name`, `entry.query`, `entry.comment`, and the first 50 chars of `entry.query` used in the subject line.
