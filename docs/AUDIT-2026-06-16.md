# Security & Code Audit — 2026-06-16

**Repository:** `matthh/transcript-app` (fork of `jbennygold/transcript-app`)
**Reviewer:** automated weekly audit
**Scope:** security, performance, maintainability
**Note:** This is a fork. Fixes cannot be applied to source files; all findings
are for upstream awareness and future fork decisions.

---

## Critical

### C-1 — `/api/rebuild` is unauthenticated and triggers CI with a PAT
**File:** `src/app/api/rebuild/route.ts`
**Severity:** Critical

`POST /api/rebuild` dispatches a GitHub Actions workflow using `GITHUB_PAT`
with no auth check on the route itself. Any unauthenticated HTTP client that
knows the URL can trigger arbitrary CI runs against the upstream repository.

**Recommended fix:** Add a secret bearer token check (e.g. `REBUILD_SECRET`)
before processing the request, identical in pattern to the `/api/warmup` token
check.

---

### C-2 — AssemblyAI webhook accepts unverified payloads
**File:** `src/app/api/transcribe/webhook/route.ts`
**Severity:** Critical

The webhook handler verifies only that `transcript_id` exists in Blob storage.
It does not verify an HMAC signature from AssemblyAI. An attacker who can guess
or enumerate a job ID can POST a fake "completed" payload that triggers loading
arbitrary data from AssemblyAI's API (job must exist there too), but more
immediately can mark any in-flight job as "failed" or cause it to attempt
retrieval of a different transcript ID's data — potentially corrupting stored
transcripts.

**Recommended fix:** Validate the `Authorization` header that AssemblyAI sends
with webhooks (Bearer token set in the `webhook_auth_header_name` / `webhook_auth_header_value`
fields at submission time).

## High

### H-1 — `/api/debug` is unauthenticated and leaks internal state
**File:** `src/app/api/debug/route.ts`
**Severity:** High

`POST /api/debug` performs a live Haiku LLM classification of the supplied
query and returns the full internal `FilterSpec`, all episode metadata, and
direct film search results — all without any authentication. This exposes
query-planner internals and incurs API costs for any caller.

**Recommended fix:** Gate with `WARMUP_TOKEN` or a dedicated `DEBUG_SECRET`
env var, or remove the route from production builds.

---

### H-2 — HTML injection in transcription-error email
**File:** `src/app/api/transcription-error/route.ts` (lines ~85–150)
**Severity:** High

`episodeTitle`, `reporterName`, `selectedText`, `correctedText`, and
`originalText` are interpolated directly into an HTML string that is sent via
Resend. No HTML-entity encoding is applied. A user who submits a crafted
transcription error report can inject arbitrary HTML (including script tags if
the email client renders them) into the operator's inbox.

**Recommended fix:** Escape `<`, `>`, `&`, `"`, `'` in all user-supplied
fields before interpolation, or use a templating library that auto-escapes.

---

### H-3 — PodReview password comparison is not timing-safe
**File:** `src/lib/podreview-auth.ts` (line 5)
**Severity:** High

`auth.slice(7) === process.env.PODREVIEW_PASSWORD` is a direct string
comparison, which short-circuits on the first differing byte. This allows
timing-based brute-force attacks to enumerate the correct password character
by character. There is also no rate limiting on the PodReview auth endpoints.

**Recommended fix:** Use `crypto.timingSafeEqual` (already imported in
`external-auth.ts`) and add request rate limiting.

---

### H-4 — Feedback and query-log Blobs are publicly readable
**Files:** `src/lib/query-logger.ts`, `src/app/api/feedback/route.ts`,
`src/app/api/cleanup-feedback/route.ts`, `src/app/api/eval/feedback/route.ts`
**Severity:** High

All four logging paths write to Vercel Blob with `access: 'public'`. The
pathnames are predictable (`query-logs/`, `feedback/`, `cleanup-feedback/`,
`eval-feedback/`). Anyone who discovers a blob URL (e.g. via the Vercel Blob
list API or logs) can read full user queries, names, and answers.

**Recommended fix:** Use `access: 'private'` (Vercel Blob private blobs
require a signed token) for all log/feedback writes, or store logs in a
proper database.

---

### H-5 — External API rate limiter is per-Lambda-instance (not global)
**File:** `src/lib/external-rate-limit.ts` (comment at top of file)
**Severity:** High

The in-memory `Map` of rate-limit buckets is scoped to a single serverless
instance. Under a Vercel deployment spanning multiple concurrent Lambda
instances (common under load), each instance has a separate counter — so a
caller can exceed the stated limit by a factor equal to the number of active
instances. The code itself notes this limitation.

**Recommended fix:** Swap to Upstash Redis or Vercel KV for globally-consistent
counters across instances.

## Medium

### M-1 — Google Sheet ID and GitHub repo hardcoded as literals
**Files:** `src/app/api/podreview/update-pdc/route.ts` (line 6),
`src/app/api/rebuild/route.ts` (line 4–5)
**Severity:** Medium

`SHEET_ID = '1RbtWP966CxA57PatyMcbJ6_ylg9tmCvDSLNxjmQ4bvk'` and
`GITHUB_REPO = 'jbennygold/transcript-app'` are string literals. Renaming
the repo or moving the sheet silently breaks these routes with no obvious
failure message.

**Recommended fix:** Move to `GOOGLE_SHEET_ID` and `GITHUB_REPO` env vars.

---

### M-2 — Anthropic client instantiated ad-hoc in two routes
**Files:** `src/app/api/cleanup-transcript/route.ts` (line 17),
`src/app/api/detect-samples/route.ts` (line 11)
**Severity:** Medium

`const anthropic = new Anthropic()` is called at module-load time directly
(not via `getAnthropic()` singleton from `lib/claude.ts`). This bypasses the
singleton pattern and creates extra client instances with their own connection
pools.

**Recommended fix:** Import and call `getAnthropic()` from `lib/claude.ts`.

---

### M-3 — Vector store and BM25 index are never refreshed in a warm Lambda
**Files:** `src/lib/vectorstore.ts`, `src/lib/bm25-loader.ts`
**Severity:** Medium

Both modules cache their data in module-level variables. A warm Lambda instance
running for hours will serve stale search data after a re-ingest until the
instance is recycled by a Vercel deployment. There is no TTL or invalidation
mechanism.

**Recommended fix:** Add a blob ETag / last-modified header check on each
request (cheap) and reload only when the index changes. Alternatively, trigger
a deployment on every ingest.

---

### M-4 — `/api/transcripts` lists and loads all transcripts on every request
**File:** `src/app/api/transcripts/route.ts`
**Severity:** Medium

The `GET /api/transcripts` handler calls `listBlobTranscripts()` and then
`loadBlobTranscript()` for every transcript on every request to build the
metadata list. With 300+ episodes this is an O(N) serial Blob-fetch waterfall
per request. The Blob reads inside the `for` loop are not parallelised.

**Recommended fix:** Parallelize with `Promise.all`, and cache the result
(e.g. a short-lived in-memory TTL or a pre-built manifest blob).

---

### M-5 — `speakers` route loads all transcripts on every request
**File:** `src/app/api/speakers/route.ts`
**Severity:** Medium

Same pattern as M-4: lists all transcripts and loads each one serially to
aggregate speaker counts. No caching.

**Recommended fix:** Same as M-4 — parallelise and cache.

---

### M-6 — TMDB API key sent as query-string parameter
**Files:** `src/app/api/kev/route.ts` (line ~57), `src/app/api/podreview/tmdb-search/route.ts`
**Severity:** Medium

`api_key=${apiKey}` is appended to TMDB search URLs as a query parameter.
While these are server-to-server calls, the key appears in TMDB server logs
and in any request-tracing middleware. TMDB recommends the `Authorization:
Bearer` header instead.

**Recommended fix:** Use `Authorization: Bearer ${apiKey}` header and remove
the `api_key` query param.

---

### M-7 — PodReview submit route is a stub that does not write to the sheet
**File:** `src/app/api/podreview/submit/route.ts`
**Severity:** Medium

The route returns `200 ok` with a "logged successfully, sheet write not yet
enabled" message. This creates a silent no-op for any UI component that
assumes the data was persisted. There is no error or warning surfaced to the
user.

**Recommended fix:** Either wire up the actual write path (like `update-pdc`
does) or return a `501 Not Implemented` so callers can detect the gap.

---

### M-8 — No query length limit on internal `/api/search` routes
**File:** `src/app/api/search/route.ts`, `src/app/api/search/stream/route.ts`
**Severity:** Medium

The external API correctly enforces `MAX_QUERY_LENGTH = 2000`. The internal
`/api/search` and `/api/search/stream` routes have no such cap, allowing
arbitrarily long queries to reach the LLM classifier and synthesis steps,
which could inflate costs and latency.

**Recommended fix:** Apply the same 2000-character cap to all search routes.

## Low

### L-1 — ChromaDB dependency present but unused in production
**Files:** `src/lib/chroma.ts`, `package.json`
**Severity:** Low

The `chromadb` npm package is in `dependencies` (not `devDependencies`) and is
listed in `next.config.js` as a `serverExternalPackage`. It is only used by
older local-dev scripts. It adds unnecessary bundle weight and a potential
supply-chain surface.

**Recommended fix:** Move to `devDependencies` or remove if the local scripts
are no longer needed.

---

### L-2 — `googleapis` in production dependencies for a route used only by admins
**Files:** `src/app/api/podreview/update-pdc/route.ts`, `package.json`
**Severity:** Low

`googleapis` is a large package (~5 MB) in `dependencies`. It is only used by
the PodReview update-pdc route, which is an internal admin-only tool. It is
listed in `serverExternalPackages` so it is not bundled, but it adds
installation overhead and is another supply-chain surface.

**Recommended fix:** Acceptable as-is given `serverExternalPackages`; consider
auditing for transitive vulnerability alerts.

---

### L-3 — Hardcoded Resend fallback address
**File:** `src/app/api/transcription-error/route.ts` (line 30),
`src/app/api/feedback/route.ts`
**Severity:** Low

`const FEEDBACK_EMAIL = process.env.FEEDBACK_EMAIL || 'delivered@resend.dev'`
means missing env config silently sends emails to Resend's catch-all test
address instead of failing visibly.

**Recommended fix:** Throw or log a clear error when `FEEDBACK_EMAIL` is not
configured.

---

### L-4 — `ab-intents-2026-02-12.log` committed to repo root
**File:** `ab-intents-2026-02-12.log`
**Severity:** Low

A large log file from an A/B experiment is committed to the repo root. This
bloats the repository and may contain user query strings.

**Recommended fix:** Add `*.log` to `.gitignore` and remove the file from
history (`git filter-repo` or BFG).

---

### L-5 — `planv3.md`, `planv4.md`, `planv4-critique.md`, `rewrite-critique.md` in repo root
**File:** repo root
**Severity:** Low

Multiple planning and critique markdown files are in the repo root alongside
application code. They reference internal decisions and implementation details
that may be sensitive (API strategies, cost estimates, etc.).

**Recommended fix:** Move to `docs/decisions/` or remove before public release.
