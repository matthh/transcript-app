# Security & Code Audit — 2026-06-23

**Repository:** `matthh/transcript-app` (fork of `jbennygold/transcript-app`)
**Reviewer:** automated weekly audit
**Scope:** security, performance, maintainability
**Note:** This is a fork. Fixes cannot be applied to source files; all findings
are for upstream awareness and future fork decisions.

**Changes since 2026-06-16 audit:** No new source commits were detected in the
fork's shallow clone. All findings from last week's audit remain open. This
report covers new findings from a fresh full-codebase read, including recently
added routes (`/api/tilda`, `/api/playlist`, `/api/speakers`) and modules
(`lib/share-summary.ts`, `lib/use-case-classifier.ts`, `lib/search-tuning.ts`).

---

## Critical

### C-1 — `/api/rebuild` is unauthenticated and triggers CI with a PAT
**File:** `src/app/api/rebuild/route.ts`
**Status:** Open (carried from 2026-06-16)

`POST /api/rebuild` dispatches a GitHub Actions workflow using `GITHUB_PAT`
with no auth check on the route itself. Any unauthenticated HTTP client that
knows the URL can trigger arbitrary CI runs against the upstream repository.

**Recommended fix:** Add a secret bearer token check (e.g. `REBUILD_SECRET`)
before processing the request, identical in pattern to the `/api/warmup` token
check.

---

### C-2 — AssemblyAI webhook accepts unverified payloads
**File:** `src/app/api/transcribe/webhook/route.ts`
**Status:** Open (carried from 2026-06-16)

The webhook handler verifies only that `transcript_id` exists in Blob storage.
It does not verify an HMAC/bearer signature from AssemblyAI. An attacker who
can guess or enumerate a job ID can POST a fake "completed" payload, triggering
transcript retrieval and overwrite — or mark any in-flight job as "failed",
corrupting stored transcripts.

**Recommended fix:** Validate the `Authorization` header that AssemblyAI sends
with webhooks (set via `webhook_auth_header_name` / `webhook_auth_header_value`
fields at submission time).

## High

### H-1 — `/api/debug` is unauthenticated and leaks internal state
**File:** `src/app/api/debug/route.ts`
**Status:** Open (carried from 2026-06-16)

`POST /api/debug` performs a live Haiku LLM classification of the supplied
query and returns the full internal `FilterSpec`, all episode metadata, and
direct film search results — without any authentication. It also contains a
hardcoded debugging artifact (`filmFilter = 'close encounters'` on line 18)
that is always evaluated regardless of the query.

**Recommended fix:** Gate with `WARMUP_TOKEN` or a dedicated `DEBUG_SECRET`
env var, or remove the route from production builds. Remove the hardcoded
`filmFilter` artifact.

---

### H-2 — HTML injection in transcription-error email
**File:** `src/app/api/transcription-error/route.ts` (lines ~97–160)
**Status:** Open (carried from 2026-06-16)

`episodeTitle`, `reporterName`, `selectedText`, `correctedText`, and
`originalText` are interpolated directly into an HTML string sent via Resend.
No HTML-entity encoding is applied. A user who submits a crafted transcription
error report can inject arbitrary HTML into the operator's inbox.

**Recommended fix:** Escape `<`, `>`, `&`, `"`, `'` in all user-supplied
fields before interpolation, or use a templating library that auto-escapes.

---

### H-3 — PodReview password comparison is not timing-safe
**File:** `src/lib/podreview-auth.ts` (line 6)
**Status:** Open (carried from 2026-06-16)

`auth.slice(7) === process.env.PODREVIEW_PASSWORD` is a direct string
comparison, which short-circuits on the first differing byte. This allows
timing-based brute-force attacks to enumerate the correct password character
by character. There is also no rate limiting on the PodReview auth endpoints.

**Recommended fix:** Use `crypto.timingSafeEqual` (already imported in
`external-auth.ts`) and add request rate limiting.

---

### H-4 — Feedback and query-log Blobs are publicly readable
**Files:** `src/lib/query-logger.ts`, `src/app/api/feedback/route.ts`,
`src/app/api/cleanup-feedback/route.ts`, `src/app/api/eval/feedback/route.ts`
**Status:** Open (carried from 2026-06-16)

All four logging paths write to Vercel Blob with `access: 'public'`. The
pathnames are predictable (`query-log/`, `feedback/`, `cleanup-feedback/`,
`eval-feedback/`). Anyone who discovers a blob URL can read full user queries,
names, and answers.

**Recommended fix:** Use `access: 'private'` (Vercel Blob private blobs
require a signed token) for all log/feedback writes, or store logs in a
proper database.

---

### H-5 — External API rate limiter is per-Lambda-instance (not global)
**File:** `src/lib/external-rate-limit.ts`
**Status:** Open (carried from 2026-06-16)

The in-memory `Map` of rate-limit buckets is scoped to a single serverless
instance. Under a Vercel deployment spanning multiple concurrent Lambda
instances, each instance has a separate counter — so a caller can exceed the
stated limit by a factor equal to the number of active instances.

**Recommended fix:** Swap to Upstash Redis or Vercel KV for globally-consistent
counters across instances.

---

### H-6 — `/api/tilda` instantiates a new Anthropic client per request and leaks TMDB key in query string
**File:** `src/app/api/tilda/route.ts` (lines 77, 53)
**Status:** New

`generateTilda()` calls `new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY })`
on every invocation (not using the `getAnthropic()` singleton from `lib/claude.ts`),
wasting connection-pool resources. Additionally, `fetchTmdbCast()` appends
`api_key=${apiKey}` to TMDB URLs as a query parameter on both the search and
credits endpoints (lines 53, 61), exposing the key in TMDB access logs.

**Recommended fix:** Replace `new Anthropic(...)` with `getAnthropic()` from
`lib/claude.ts`. Replace `api_key` query params with `Authorization: Bearer`
header per TMDB's recommendation.

## Medium

### M-1 — Google Sheet ID and GitHub repo hardcoded as literals
**Files:** `src/app/api/podreview/update-pdc/route.ts` (line 7),
`src/app/api/rebuild/route.ts` (line 3)
**Status:** Open (carried from 2026-06-16)

`SHEET_ID = '1RbtWP966CxA57PatyMcbJ6_ylg9tmCvDSLNxjmQ4bvk'` and
`GITHUB_REPO = 'jbennygold/transcript-app'` are string literals. Renaming
the repo or moving the sheet silently breaks these routes.

**Recommended fix:** Move to `GOOGLE_SHEET_ID` and `GITHUB_REPO` env vars.

---

### M-2 — Anthropic client instantiated ad-hoc in multiple modules
**Files:** `src/app/api/cleanup-transcript/route.ts` (line 17),
`src/app/api/detect-samples/route.ts` (line 8),
`src/lib/share-summary.ts` (line 22)
**Status:** New / Open

Three modules construct `new Anthropic()` directly instead of using the
`getAnthropic()` singleton from `lib/claude.ts`. `cleanup-transcript` and
`detect-samples` do so at module-load time (incurred per Lambda cold start);
`share-summary.ts` does it per call.

**Recommended fix:** Import and call `getAnthropic()` from `lib/claude.ts` in
all three files.

---

### M-3 — Vector store, BM25 index, and playlist data are never refreshed in a warm Lambda
**Files:** `src/lib/vectorstore.ts`, `src/lib/bm25-loader.ts`,
`src/app/api/playlist/route.ts`
**Status:** New / Open (playlist; vector/BM25 carried from 2026-06-16)

All three modules cache data in module-level variables. A warm Lambda instance
running for hours will serve stale search data or stale playlist data after a
re-ingest or playlist rebuild until the instance is recycled.

**Recommended fix:** Add a blob ETag / last-modified header check on each
request (cheap) and reload only when the index changes. Alternatively, trigger
a deployment on every ingest.

---

### M-4 — `/api/transcripts` loads all transcripts serially on every request
**File:** `src/app/api/transcripts/route.ts`
**Status:** Open (carried from 2026-06-16)

The `GET /api/transcripts` handler calls `listBlobTranscripts()` and then
`loadBlobTranscript()` for every transcript sequentially, with no
parallelisation and no caching.

**Recommended fix:** Parallelise with `Promise.all`, and cache the result
(e.g. a short-lived in-memory TTL or a pre-built manifest blob).

---

### M-5 — `/api/speakers` is an O(N) serial Blob-fetch waterfall with no caching
**File:** `src/app/api/speakers/route.ts` (lines 12–29)
**Status:** New

Same pattern as M-4: lists all transcripts and loads each one sequentially
from Vercel Blob to aggregate speaker counts. With 300+ transcripts this is an
expensive waterfall on every call, and the result is never cached.

**Recommended fix:** Parallelise with `Promise.all`. Cache the result or
pre-build a speaker manifest blob during ingest.

---

### M-6 — TMDB API key sent as query-string parameter
**Files:** `src/app/api/kev/route.ts`, `src/app/api/podreview/tmdb-search/route.ts`,
`src/app/api/tilda/route.ts` (lines 53, 61)
**Status:** Open (carried from 2026-06-16; `tilda` route is new)

`api_key=${apiKey}` is appended to TMDB search and credits URLs. While
server-to-server, the key appears in TMDB server logs and request traces.

**Recommended fix:** Use `Authorization: Bearer ${apiKey}` header and remove
the `api_key` query param across all three files.

---

### M-7 — PodReview submit route is a stub that does not write to the sheet
**File:** `src/app/api/podreview/submit/route.ts`
**Status:** Open (carried from 2026-06-16)

The route returns `200 ok` with a "logged successfully, sheet write not yet
enabled" message, creating a silent no-op for any UI component that assumes
data was persisted.

**Recommended fix:** Either wire up the actual write path or return `501 Not
Implemented` so callers can detect the gap.

---

### M-8 — No query length limit on internal `/api/search` routes
**File:** `src/app/api/search/route.ts`, `src/app/api/search/stream/route.ts`
**Status:** Open (carried from 2026-06-16)

The external API correctly enforces `MAX_QUERY_LENGTH = 2000`. The internal
search routes have no such cap.

**Recommended fix:** Apply the same 2000-character cap to all search routes.

---

### M-9 — Agent search loads all transcripts into Lambda memory on first call
**File:** `src/lib/agent-search.ts` (`loadAllTranscripts()`, line 43)
**Status:** New

On the first agent search invocation, `loadAllTranscripts()` fetches all 300+
transcript blobs into a module-level `Map`. This can cause significant cold-
start latency (serial Blob fetches behind the `listBlobTranscripts()` call) and
high peak Lambda memory usage. The cache is never TTL-expired or size-bounded.

**Recommended fix:** Pre-build a searchable index of transcript content during
ingest (e.g. a single aggregated blob) so agent search loads one file rather
than 300+. Alternatively, load transcripts lazily on demand rather than all-at-once.

---

### M-10 — `search-tuning.ts` fast variant defaults to a legacy Haiku model ID
**File:** `src/lib/search-tuning.ts` (line 22)
**Status:** New

`getSearchTuning('fast')` defaults to `'claude-3-haiku-20240307'` when
`INTERPRETIVE_FAST_MODEL` is unset. The rest of the codebase has migrated to
`claude-haiku-4-5-20251001`. When the env var is not explicitly set, the fast
variant silently uses the older model.

**Recommended fix:** Change the default to `'claude-haiku-4-5-20251001'` to
match the rest of the codebase, or add a deprecation log when the old model ID
is used.

### L-7 — `/api/tilda` is unauthenticated and incurs LLM + TMDB cost per call
**File:** `src/app/api/tilda/route.ts`
**Status:** New

When no metadata match is found, `GET /api/tilda?film=…` calls TMDB and then
Claude Haiku to generate answers — with no authentication, no rate limiting,
and no caching of generated results. An automated client enumerating film names
could drive up API costs.

**Recommended fix:** Add a simple token gate, a per-IP rate limit, or cache
generated results in Vercel Blob/KV to avoid repeat LLM calls for the same film.
